title: "Multi-GPU inference"
description: "Scale IPEX-LLM across multiple Intel Arc GPUs using pipeline parallelism, FlashMoE, or tensor parallelism via vLLM."
---

IPEX-LLM supports three approaches to use multiple Intel GPUs for LLM inference:

- **Pipeline parallel inference** — split model layers vertically across GPUs using the `transformers` API and `torchrun`
- **FlashMoE** — run 671B MoE models (DeepSeek V3/R1) or 235B MoE models (Qwen3MoE) on 1–2 Arc GPUs using the llama.cpp portable zip
- **Tensor parallel via vLLM** — serve multiple users at high throughput with `--tensor-parallel-size N` in the IPEX-LLM vLLM backend

---

### Select specific GPUs

**In Python**, target a specific GPU by index:

```python
model = model.to('xpu:1')
input_ids = tokenizer.encode(prompt, return_tensors="pt").to('xpu:1')
```

**With an environment variable**, use `ONEAPI_DEVICE_SELECTOR` to restrict which GPUs are visible to your process:

<CodeGroup>
```bash Linux — second GPU only
export ONEAPI_DEVICE_SELECTOR=level_zero:1
python generate.py
```

```cmd Windows — second GPU only
set ONEAPI_DEVICE_SELECTOR=level_zero:1
python generate.py
```
</CodeGroup>

To use both GPU 0 and GPU 1:

```bash
export ONEAPI_DEVICE_SELECTOR="level_zero:0;level_zero:1"
```

## Pipeline parallel inference

Pipeline parallelism distributes model layers vertically across GPUs. This allows models that are too large for a single GPU's VRAM to run across two or more cards.

### Requirements

- Linux (tested on Intel Arc A770, Data Center GPU Max)
- At least two Intel Arc or Data Center GPUs
- Intel® oneAPI Base Toolkit 2024.0 with GPU driver installed

### Installation

```bash
conda create -n llm python=3.11
conda activate llm

pip install --pre --upgrade ipex-llm[xpu] \
  --extra-index-url https://pytorch-extension.intel.com/release-whl/stable/xpu/us/

pip install oneccl_bind_pt==2.1.100 \
  --index-url https://pytorch-extension.intel.com/release-whl/stable/xpu/us/
```

### Run pipeline parallel inference

Use `torchrun` to launch one process per GPU. The `pipeline_parallel_stages` parameter in `from_pretrained` controls how many GPUs the model is split across.

```python
from ipex_llm.transformers import AutoModelForCausalLM, init_pipeline_parallel
from transformers import AutoTokenizer
import torch

init_pipeline_parallel()

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-13b-chat-hf",
    load_in_low_bit="sym_int4",
    optimize_model=True,
    trust_remote_code=True,
    use_cache=True,
    torch_dtype=torch.float16,
    pipeline_parallel_stages=2,  # number of GPUs
)

tokenizer = AutoTokenizer.from_pretrained(
    "meta-llama/Llama-2-13b-chat-hf",
    trust_remote_code=True,
)

local_rank = torch.distributed.get_rank()

with torch.inference_mode():
    input_ids = tokenizer.encode(
        "Once upon a time,", return_tensors="pt"
    ).to(f"xpu:{local_rank}")

    output = model.generate(input_ids, max_new_tokens=32)

    if local_rank == 1:  # last GPU prints result
        print(tokenizer.decode(output[0], skip_special_tokens=True))
```

### Launch command

Set the recommended environment variables and launch with `torchrun`:

```bash
source /opt/intel/oneapi/setvars.sh

export MASTER_ADDR=127.0.0.1
export MASTER_PORT=9090
export FI_PROVIDER=tcp
export USE_XETLA=OFF
export OMP_NUM_THREADS=6
export IPEX_LLM_QUANTIZE_KV_CACHE=1
export SYCL_PI_LEVEL_ZERO_USE_IMMEDIATE_COMMANDLISTS=1
export TORCH_LLM_ALLREDUCE=0

NUM_GPUS=2

CCL_ZE_IPC_EXCHANGE=sockets torchrun \
  --standalone \
  --nnodes=1 \
  --nproc-per-node $NUM_GPUS \
  generate.py \
  --repo-id-or-model-path 'meta-llama/Llama-2-7b-chat-hf' \
  --gpu-num $NUM_GPUS \
  --low-bit 'sym_int4'
```

<Note>
INT4 (`sym_int4`) is applied by default. You can specify other low-bit formats such as `fp8` or `fp6` via `--low-bit`.
</Note>

## FlashMoE for 671B and 235B MoE models

FlashMoE is a specialized tool that keeps the enormous weight matrices of MoE models in CPU memory and dispatches active expert weights to one or more Intel Arc GPUs on demand. This makes it possible to run **DeepSeek V3/R1 671B** and **Qwen3MoE 235B** on commodity hardware.

<Warning>
FlashMoE is currently **Linux only** and uses the llama.cpp portable zip. Do not source a oneAPI environment before running it.
</Warning>

### Hardware requirements

| Model | CPU memory | GPU | Disk |
|---|---|---|---|
| DeepSeek V3/R1 671B (INT4) | 380 GB | 1–8 Arc A770 or B580 | 500 GB |
| Qwen3MoE 235B (INT4) | 128 GB | 1–8 Arc A770 or B580 | 500 GB |

<Note>
With a single Arc A770, reduce the context size to avoid OOM. Add `-c 1024` to the command.
</Note>

### Run DeepSeek R1 671B with FlashMoE

Download `DeepSeek-R1-Q4_K_M` from [unsloth/DeepSeek-R1-GGUF](https://huggingface.co/unsloth/DeepSeek-R1-GGUF/tree/main/DeepSeek-R1-Q4_K_M), then extract the llama.cpp portable zip and run:

```bash
# CLI inference
./flash-moe -m /PATH/TO/DeepSeek-R1-Q4_K_M-00001-of-00009.gguf \
  --prompt "What's AI?" -no-cnv
```

```bash
# HTTP server (OpenAI-compatible)
./flash-moe -m /PATH/TO/DeepSeek-R1-Q4_K_M-00001-of-00009.gguf \
  --serve -n 512 -np 2 -c 4096
```

The server starts at `http://127.0.0.1:8080`.

<Tip>
On dual-socket Xeon systems, enable **SNC (Sub-NUMA Clustering)** in BIOS and prefix your command with `numactl --interleave=all` to improve decoding throughput.
</Tip>

## Tensor parallel via vLLM

For high-throughput serving workloads, the IPEX-LLM vLLM backend supports tensor parallelism across multiple GPUs. Each GPU holds a shard of the attention heads and MLP layers, enabling larger batch sizes and higher token throughput.

Set `--tensor-parallel-size` to the number of GPUs you want to use:

```bash
python -m ipex_llm.vllm.xpu.entrypoints.openai.api_server \
  --port 8000 \
  --model "/llm/models/Qwen2.5-7B-Instruct/" \
  --served-model-name "Qwen2.5-7B-Instruct" \
  --device xpu \
  --dtype float16 \
  --enforce-eager \
  --load-in-low-bit fp8 \
  --tensor-parallel-size 4 \
  --distributed-executor-backend ray \
  --gpu-memory-utilization 0.95 \
  --max-model-len 2000
```

<Note>
For BMG GPUs, set `CCL_WORKER_COUNT=1`. For other Arc GPUs, use `CCL_WORKER_COUNT=2`.
</Note>

See the [Docker guide](/deployment/docker) for the full vLLM container setup, including all required environment variables and the complete `docker run` command.

## Memory considerations

- **Pipeline parallel**: model layers are split sequentially. GPU 0 holds early layers, GPU N-1 holds later layers. Each GPU needs enough VRAM for its share of the model plus KV cache and activations.
- **Tensor parallel**: each GPU holds a column/row shard of every layer. Memory is divided more evenly, but requires fast interconnect between GPUs (PCIe or equivalent).
- **FlashMoE**: most weights live in CPU RAM. GPUs are used for compute only, so VRAM requirements are low even for 671B models. Performance depends heavily on CPU–GPU bandwidth.

<Tip>
To quantify KV cache memory, set `IPEX_LLM_QUANTIZE_KV_CACHE=1` before running pipeline parallel inference. This compresses the KV cache to INT8, reducing per-GPU VRAM usage at a small quality cost.
</Tip>
